## 5. 经验教训

### 5.1 成功经验
1. **真实 arXiv ID 验证**：提前给出已验证 ID 列表（2305.18290、2203.02155 等），避免编造
2. **Pyodide 兼容性**：每条代码必须明确标注可运行环境
3. **hedging 语言**：具体数字必须带"据公开报道"/"行业估算"
4. **公式不重复**：fenced code block 与 inline 公式二选一
5. **5 轮审计一次过**：L25 用 5 轮审计换来了深度修复（fabrication/UI 各修复到位），后续 L26/L27/L28 各 1 轮就过

### 5.2 待改进
1. **图片资源**：lora.svg 等新课配图未创建（建议下个阶段统一生成）
2. **自动化流水线缺少 fail-fast**：should 在 Plan 阶段就检查数据，避免 Generate 失败后才发现
3. **polish 阶段 cross-link 不够多**：建议 ≥3 个

### 7.2 阶段验收

按 W2-W4 同样的 7 阶段管线 + 双重审计。**重大差异**：W2-W4 主要补 LLM/Agent 微调，W5-W8 转向**推理 + 分布式 + RAG深化 + 多模态**，挑战更大。

### 7.3 风险预警
- **L29 vLLM**：与 L25 重叠（都是推理部署），需差异化
- **L30 分布式**：需要真实 DDP/FSDP/DeepSpeed 代码示例，环境要求最高
- **L31 17 种策略**：内容密度大，可能超过章节字数限制（建议拆成 2 部分）
- **L32 多模态**：Pyodide 不支持图像处理，需明确降级到 numpy 模拟
